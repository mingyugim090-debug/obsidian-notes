## 1. API 기초

  

### API란?

> [!note] 정의

> **API (Application Programming Interface)** — 소프트웨어끼리 통신하는 규칙의 집합.

> 클라이언트(프론트엔드)가 서버(백엔드)에게 데이터를 요청하고 받는 방식.



```

클라이언트 → 요청 (Request) → 서버

클라이언트 ← 응답 (Response) ← 서버

```

  

### REST API


**RE**presentational **S**tate **T**ransfer — 웹에서 가장 많이 쓰이는 API 설계 방식.

  

| HTTP 메서드 | 의미 | 예시 |

|------------|------|------|

| `GET` | 데이터 조회 | `GET /api/customers` |

| `POST` | 데이터 생성 | `POST /api/customers` |

| `PUT` | 데이터 전체 수정 | `PUT /api/customers/123` |

| `PATCH` | 데이터 일부 수정 | `PATCH /api/customers/123` |

| `DELETE` | 데이터 삭제 | `DELETE /api/customers/123` |

  

### HTTP 상태 코드

  

| 코드 | 의미 | 상황 |

|------|------|------|

| `200` | OK | 성공 |

| `201` | Created | 데이터 생성 성공 |

| `400` | Bad Request | 잘못된 요청 (클라이언트 오류) |

| `401` | Unauthorized | 인증 안 됨 (로그인 필요) |

| `403` | Forbidden | 권한 없음 (로그인은 됐지만 접근 불가) |

| `404` | Not Found | 리소스 없음 |

| `429` | Too Many Requests | 요청 횟수 초과 (Rate Limit) |

| `500` | Internal Server Error | 서버 내부 오류 |

| `502` | Bad Gateway | 서버 간 통신 오류 |

  

### 요청 구조

  

```http

POST /api/customers HTTP/1.1

Host: hairflow.vercel.app

Content-Type: application/json ← Header: 데이터 형식 선언

Authorization: Bearer eyJhbGc... ← Header: 인증 토큰

  

{ ← Body: 전송 데이터

"name": "김민재",

"phone": "010-1234-5678"

}

```

  

### API 응답 통일 형식 (권장)

  

```typescript

// 성공

{ data: T, error: null }

  

// 실패

{ data: null, error: { code: string, message: string } }

```

## 2. 인증 & 보안

  

### OAuth 2.0

  

> [!note] 정의

> **OAuth** — 사용자가 비밀번호를 공유하지 않고 제3자 서비스(카카오, 구글)를 통해 로그인하는 표준 프로토콜.

  

```

사용자 → 내 앱 → "카카오로 로그인" 클릭

→ 카카오 로그인 페이지로 리다이렉트

→ 카카오에서 인증 완료

→ 내 앱으로 code 전달 (callback URL)

→ code로 access_token 요청

→ access_token으로 사용자 정보 조회

```

  

**주요 개념**

  

| 용어 | 설명 |

|------|------|

| `client_id` | 내 앱 식별자 (카카오/구글 개발자 콘솔에서 발급) |

| `redirect_uri` | 인증 후 돌아올 URL (예: `/api/auth/callback`) |

| `access_token` | 사용자 정보 조회에 사용하는 임시 열쇠 (수명 짧음) |

| `refresh_token` | access_token 재발급용 (수명 긺) |

| `scope` | 요청하는 권한 범위 (예: `email`, `profile`) |

  

### JWT (JSON Web Token)

  

> [!note] 정의

> 서버가 사용자 정보를 암호화하여 클라이언트에게 주는 토큰. 서버는 DB를 조회하지 않고 토큰 자체로 사용자를 식별.

  

```

eyJhbGciOiJIUzI1NiJ9 . eyJ1c2VyX2lkIjoiMTIzIn0 . 서명

↑헤더 ↑페이로드(데이터) ↑변조 방지

```

  

- **페이로드**: `user_id`, `email`, `role`, 만료시간(`exp`) 등 포함

- **검증**: 서버는 서명만 확인 → DB 조회 불필요 → 빠름

- **주의**: 페이로드는 Base64 디코딩으로 누구나 읽을 수 있음 → 민감 정보 넣지 말 것

  

### Session vs Token 인증

  

| 구분 | Session | JWT Token |

|------|---------|-----------|

| 저장 위치 | 서버 메모리/DB | 클라이언트 (쿠키/LocalStorage) |

| 확장성 | 서버 여러 대면 공유 어려움 | 서버 상태 없음 (Stateless) |

| 보안 | 서버 제어 가능 | 토큰 탈취 시 만료 전까지 사용 가능 |

| Supabase 방식 | Session + JWT 혼용 | — |

  

### Cookie vs LocalStorage

  

| 구분 | Cookie | LocalStorage |

|------|--------|-------------|

| 수명 | 만료일 설정 가능 | 브라우저 닫아도 유지 |

| 용량 | 4KB | 5-10MB |

| 서버 전송 | 자동 전송 (HTTP 헤더) | 수동으로 포함해야 함 |

| XSS 공격 | `HttpOnly` 설정 시 JS 접근 불가 (안전) | JS로 접근 가능 (위험) |

| 권장 용도 | 인증 토큰 | 비민감 UI 상태 |

  

### RLS (Row Level Security)

  

> [!important] Supabase 핵심 보안 개념

> 데이터베이스 테이블의 각 행(Row)에 접근 권한을 설정하는 PostgreSQL 기능.

> "같은 가족만 같은 가족의 데이터를 볼 수 있다" 같은 규칙을 DB 레벨에서 강제.

  

```sql

-- RLS 활성화

ALTER TABLE consultations ENABLE ROW LEVEL SECURITY;

  

-- 정책: 본인 데이터만 접근 가능

CREATE POLICY "own_data_only" ON consultations

FOR ALL

USING (designer_id = auth.uid()); -- auth.uid() = 현재 로그인한 사용자 ID

```

  

> [!warning] 주의

> RLS를 우회하려면 `service_role_key`가 필요 (서버에서만 사용, 절대 클라이언트 노출 금지)

## 3. Supabase

  

> [!note] 정의

> **Supabase** — Firebase의 오픈소스 대안. PostgreSQL 기반 BaaS (Backend as a Service).

> DB + 인증 + 스토리지 + 실시간 + Edge Functions를 한 번에 제공.

  

### 핵심 구성요소

  

```

Supabase 프로젝트

├── PostgreSQL DB ← 데이터 저장

├── Auth ← 로그인/회원가입 (Google, Kakao 등)

├── Storage ← 파일 저장 (이미지, 영상)

├── Realtime ← 실시간 데이터 변경 감지

├── Edge Functions ← 서버리스 함수 (Deno 기반)

└── REST API ← 자동 생성된 API

```

  

### 클라이언트 생성

  

```typescript

// 브라우저용 (클라이언트 컴포넌트)

import { createBrowserClient } from '@supabase/ssr';

const supabase = createBrowserClient(

process.env.NEXT_PUBLIC_SUPABASE_URL!,

process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY! // 공개 키 (노출 가능)

);

  

// 서버용 (Server Component, API Route)

import { createServerClient } from '@supabase/ssr';

// cookies()로 세션 전달 필요

  

// 관리자용 (RLS 우회, 서버에서만!)

import { createClient } from '@supabase/supabase-js';

const adminSupabase = createClient(url, process.env.SUPABASE_SERVICE_ROLE_KEY!);

```

  

### 주요 쿼리 패턴

  

```typescript

// 조회

const { data, error } = await supabase

.from('customers')

.select('id, name, phone') // 컬럼 지정

.eq('designer_id', userId) // WHERE 조건

.order('created_at', { ascending: false })

.limit(20);

  

// 생성

const { data, error } = await supabase

.from('customers')

.insert({ name: '김민재', designer_id: userId })

.select() // 생성된 데이터 반환

.single(); // 단건 반환

  

// 수정

const { error } = await supabase

.from('customers')

.update({ name: '이민재' })

.eq('id', customerId);

  

// 삭제

const { error } = await supabase

.from('customers')

.delete()

.eq('id', customerId);

```

  

### Storage (파일 업로드)

  

```typescript

// 업로드

const { data, error } = await supabase.storage

.from('customer-photos') // 버킷 이름

.upload('path/file.jpg', file, {

contentType: 'image/jpeg',

upsert: false // true면 덮어쓰기

});

  

// 공개 URL 가져오기 (public 버킷)

const { data } = supabase.storage

.from('customer-photos')

.getPublicUrl('path/file.jpg');

  

// 서명된 URL (private 버킷, 임시 접근)

const { data } = await supabase.storage

.from('customer-photos')

.createSignedUrl('path/file.jpg', 300); // 300초 유효

```

  

### Auth

  

```typescript

// 구글 로그인

const { error } = await supabase.auth.signInWithOAuth({

provider: 'google',

options: { redirectTo: `${window.location.origin}/api/auth/callback` }

});

  

// 현재 세션 확인

const { data: { session } } = await supabase.auth.getSession();

const user = session?.user;

  

// 로그아웃

await supabase.auth.signOut();

  

// 세션 변경 감지

supabase.auth.onAuthStateChange((event, session) => {

if (event === 'SIGNED_IN') { /* 로그인 처리 */ }

if (event === 'SIGNED_OUT') { /* 로그아웃 처리 */ }

});

```

  

### Realtime (실시간 구독)

  

```typescript

// 테이블 변경 실시간 감지

const channel = supabase

.channel('events-changes')

.on('postgres_changes',

{ event: '*', schema: 'public', table: 'events' },

(payload) => {

console.log('변경 감지:', payload.eventType, payload.new);

}

)

.subscribe();

  

// 구독 해제 (컴포넌트 언마운트 시)

return () => supabase.removeChannel(channel);

```

  

### 키 종류 & 보안

  

| 키 | 노출 가능 여부 | 용도 |

|---|--------------|------|

| `NEXT_PUBLIC_SUPABASE_URL` | ✅ 가능 | DB 주소 |

| `NEXT_PUBLIC_SUPABASE_ANON_KEY` | ✅ 가능 | 공개 접근 (RLS 적용됨) |

| `SUPABASE_SERVICE_ROLE_KEY` | ❌ 절대 불가 | RLS 우회, 서버 전용 |

## 4. Vercel & 배포

  

> [!note] 정의

> **Vercel** — Next.js 제작사의 클라우드 배포 플랫폼. Git push만 해도 자동 빌드 & 배포.

  

### 배포 흐름

  

```

git push origin main

↓

Vercel이 GitHub 변경 감지

↓

npm run build 실행 (TypeScript 컴파일 + 번들링)

↓

빌드 성공 → 프로덕션 URL에 즉시 반영

빌드 실패 → 이전 버전 유지, 이메일/대시보드 알림

```

  

### Vercel 빌드 로그 읽는 법

  

```

✓ Compiled successfully in 15.4s ← 컴파일 성공

Running TypeScript ... ← 타입 검사 시작

Failed to compile. ← 빌드 실패 ← 여기서 에러 찾기

  

./src/lib/fal.ts:116:17

Type error: Object literal may only specify known properties

↑파일경로 ↑줄:컬럼 ↑에러 메시지

```

  

> [!tip] 빌드 오류 대응

> - 로컬에서 먼저 `npx tsc --noEmit` 으로 타입 에러 확인

> - `npm run build` 로 프로덕션 빌드 테스트 후 push

  

### Vercel 주요 개념

  

| 용어 | 설명 |

|------|------|

| **Production** | main 브랜치 → 실제 서비스 URL |

| **Preview** | PR/다른 브랜치 → 임시 URL (테스트용) |

| **Serverless Function** | API Routes → 요청 시에만 실행되는 함수 |

| **Edge Function** | CDN 엣지에서 실행 (더 빠름, 기능 제한 있음) |

| **Cold Start** | 서버리스 함수 첫 실행 시 준비 시간 (1-3초) |

| **maxDuration** | API Route 최대 실행 시간 설정 |

  

### API Route 실행 시간 설정

  

```typescript

// Next.js App Router에서 타임아웃 설정

export const maxDuration = 60; // 초 단위 (Vercel Pro: 최대 300초)

// AI 이미지 생성 같이 오래 걸리는 작업

```

  

### 환경변수 설정

  

```

Vercel 대시보드 → Project → Settings → Environment Variables

  

구분:

- Production: 실제 서비스용 (프로덕션 키)

- Preview: PR 미리보기용

⚠️ NEXT_PUBLIC_ 없는 변수 = 서버에서만 접근 가능

```

## 5. SDK

  

> [!note] 정의

> **SDK (Software Development Kit)** — API를 더 쉽게 사용할 수 있도록 만든 라이브러리 패키지.

> 직접 HTTP 요청을 보내는 대신 함수 호출로 API 사용 가능.

  

### API vs SDK 비교

  

```typescript

// Raw API 사용 (직접 HTTP 요청)

const response = await fetch('https://api.openai.com/v1/chat/completions', {

method: 'POST',

headers: {

'Authorization': `Bearer ${apiKey}`,

'Content-Type': 'application/json'

},

body: JSON.stringify({ model: 'gpt-4o', messages: [...] })

});

const data = await response.json();

  

// SDK 사용 (추상화된 함수 호출)

import OpenAI from 'openai';

const openai = new OpenAI({ apiKey });

const data = await openai.chat.completions.create({

model: 'gpt-4o',

messages: [...]

});

```

  

### 주요 SDK 목록

  

| SDK | 설치 | 용도 |

|-----|------|------|

| `@supabase/supabase-js` | `npm i @supabase/supabase-js` | Supabase DB/Auth/Storage |

| `@supabase/ssr` | `npm i @supabase/ssr` | Next.js SSR용 Supabase |

| `openai` | `npm i openai` | OpenAI GPT/DALL-E |

| `@fal-ai/client` | `npm i @fal-ai/client` | Fal.ai 이미지 생성 |

| `replicate` | `npm i replicate` | Replicate AI 모델 |

| `@toss/slash-react` | — | 토스 결제 SDK |

  

### SDK 타입 오류 대응

  

> [!warning] 자주 만나는 문제

> SDK 타입 정의가 실제 API 파라미터를 모두 커버하지 못할 때 TypeScript 에러 발생

  

```typescript

// 에러: 'face_id_plus_weight' does not exist in type...

await fal.subscribe('fal-ai/ip-adapter-face-id', {

input: { face_id_plus_weight: 0.8 } // ← 타입 정의에 없지만 API는 지원

});

  

// 해결: as any 캐스팅

// eslint-disable-next-line @typescript-eslint/no-explicit-any

await (fal.subscribe as any)('fal-ai/ip-adapter-face-id', {

input: { face_id_plus_weight: 0.8 }

});

```

## 6. Next.js 핵심 개념

  

### App Router vs Pages Router

  

| 구분 | App Router (권장) | Pages Router (구버전) |

|------|-----------------|---------------------|

| 폴더 | `app/` | `pages/` |

| 레이아웃 | `layout.tsx` | `_app.tsx`, `_document.tsx` |

| API | `app/api/route.ts` | `pages/api/route.ts` |

| SSR | Server Component 기본 | `getServerSideProps` |

  

### Server Component vs Client Component

  

```tsx

// Server Component (기본) — 서버에서 렌더링

// DB 접근, 환경변수 사용 가능 / useState, useEffect 사용 불가

async function CustomerList() {

const customers = await fetchFromDB(); // 서버에서 직접 DB 조회

return <ul>{customers.map(c => <li>{c.name}</li>)}</ul>;

}

  

// Client Component — 브라우저에서 렌더링

'use client'; // ← 이 선언 필수

function InteractiveButton() {

const [count, setCount] = useState(0); // 가능

return <button onClick={() => setCount(c => c+1)}>{count}</button>;

}

```

### Middleware (미들웨어)

  

```typescript

// middleware.ts — 모든 요청에서 실행 (인증 체크 등)

import { NextResponse } from 'next/server';

export function middleware(request: Request) {

const isLoggedIn = checkAuth(request);

if (!isLoggedIn) {

return NextResponse.redirect(new URL('/login', request.url));

}

return NextResponse.next();

}

  

export const config = {

matcher: ['/dashboard/:path*', '/customers/:path*'] // 적용할 경로
```

## 9. 스토리지 & CDN

  

### CDN (Content Delivery Network)

  

> [!note] 정의

> **CDN** — 전 세계 여러 서버에 파일을 복사해두고, 사용자에게 가장 가까운 서버에서 제공하는 네트워크.

> 한국 사용자 → 한국 CDN 서버에서 이미지 로드 → 빠름

  

```

원본 서버 (미국) → CDN 복제 → 한국 엣지 서버

↑

한국 사용자가 여기서 받음 (빠름)

```

  

### Signed URL vs Public URL

  

| 구분 | Public URL | Signed URL |

|------|-----------|------------|

| 접근 제한 | 누구나 접근 가능 | 임시 토큰 있어야 접근 |

| 만료 | 없음 | 설정한 시간 후 만료 |

| 용도 | 공개 이미지, 아이콘 | 개인 파일, 의료/법적 데이터 |

| 성능 | 빠름 (CDN 캐싱) | 느림 (매번 검증) |

  

```typescript

// Public URL (public 버킷)

supabase.storage.from('photos').getPublicUrl('file.jpg').data.publicUrl

// → https://xxx.supabase.co/storage/v1/object/public/photos/file.jpg

  

// Signed URL (5분 유효)

await supabase.storage.from('photos').createSignedUrl('file.jpg', 300)

// → https://xxx.supabase.co/storage/v1/object/sign/photos/file.jpg?token=...

```

## 11. 결제 연동 (Toss Payments)

  

### 결제 흐름

  

```

1. 클라이언트: loadTossPayments(clientKey) 초기화

2. 클라이언트: payments.requestPayment() → 결제창 열기

3. 사용자: 결제 진행

4. 성공 시: /success?paymentKey=...&orderId=...&amount=... 리다이렉트

5. 서버: POST /api/payment → 토스 서버에 결제 승인 요청

(secretKey 사용, 서버에서만!)

6. 서버: 금액 검증 후 DB 업데이트

```

  

### 보안 핵심

  

```typescript

// ⚠️ 금액 검증 필수: 클라이언트에서 보낸 금액을 그대로 믿으면 안 됨

const VALID_AMOUNTS = { basic: 19900, enterprise: 39900 };

  

export async function POST(request: Request) {

const { paymentKey, orderId, amount, plan } = await request.json();

  

// 클라이언트가 보낸 금액이 실제 금액과 일치하는지 검증

if (amount !== VALID_AMOUNTS[plan]) {

return NextResponse.json({ error: '금액 불일치' }, { status: 400 });

}

  

// 토스 서버에 승인 요청 (secretKey 사용)

const response = await fetch('https://api.tosspayments.com/v1/payments/confirm', {

method: 'POST',

headers: {

Authorization: `Basic ${Buffer.from(`${secretKey}:`).toString('base64')}`,

},

body: JSON.stringify({ paymentKey, orderId, amount })

});

}

```

## 12. 에러 & 디버깅

  

### 자주 만나는 에러 패턴

  

| 에러 | 원인 | 해결 |
